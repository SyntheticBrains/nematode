## Context

Phase 6 Tranche 5 (per [`phase6-tracking` Decision 1](../phase6-tracking/design.md)) is the platform refactor that takes the architecture-comparison sweep from the discrete grid onto a continuous-2D substrate with continuous-action control, and closes **Gate 2** (L1 plugin parity in practice). The roadmap canonical reference is `docs/roadmap.md` § Phase 6 § Continuous environment + sensory physics; the Gate 2 pass criteria (G2.a–G2.d) are pre-registered in [`phase6-tracking` design.md § Decision 6 § Gate 2](../phase6-tracking/design.md).

Current state (verified by codebase audit, 2026-06-05):

- **Discrete actions are baked in throughout.** `Action` is a 6-member `StrEnum`; `DEFAULT_ACTIONS` ([`brain/actions.py:20`](../../../packages/quantum-nematode/quantumnematode/brain/actions.py#L20)) is the active 4-action subset ([`brain/actions.py:8-31`](../../../packages/quantum-nematode/quantumnematode/brain/actions.py#L8)). `ActionData.action` carries the enum, not a vector. The env `_apply_movement` (defined on `BaseEnvironment`) hard-rejects any non-`DEFAULT_ACTIONS` set ([`env/env.py:1018`](../../../packages/quantum-nematode/quantumnematode/env/env.py#L1018)) and moves exactly one cell via `DIRECTION_MAP`.
- **No shared PPO base class.** `ClassicalBrain` is a `runtime_checkable` Protocol ([`brain/arch/_brain.py:449`](../../../packages/quantum-nematode/quantumnematode/brain/arch/_brain.py#L449)), so the `softmax → Categorical → log_prob/entropy` sampling and the PPO update are **copy-pasted independently** into MLP-PPO, LSTM-PPO, CfC-PPO, and connectome-PPO. There is no single chokepoint for action handling today.
- **The env is hard-coded.** `create_env_from_config` ([`utils/config_loader.py:2702`](../../../packages/quantum-nematode/quantumnematode/utils/config_loader.py#L2702)) unconditionally builds `DynamicForagingEnvironment`; `type EnvironmentType = DynamicForagingEnvironment` ([`env/env.py:4077`](../../../packages/quantum-nematode/quantumnematode/env/env.py#L4077)). There is no env-type discriminator in `EnvironmentConfig`. The movement methods (`_apply_movement`, `move_agent`) live on `BaseEnvironment` and are grid-coupled (`DIRECTION_MAP`, `move_agent(action: Action)` signature) — the continuous env must override them.
- **Sensing is already continuous-math.** Food/predator gradients are Euclidean + analytic ([`env/env.py:1926,1966`](../../../packages/quantum-nematode/quantumnematode/env/env.py#L1926)); temperature/oxygen are analytic fields in `(x,y)`; STAM stores deltas/scalars and is coordinate-agnostic ([`agent/stam.py:309`](../../../packages/quantum-nematode/quantumnematode/agent/stam.py#L309)). The position/state side already supports floats (`BrainParams.agent_position: tuple[float, float]`).
- **The T2 registry is clean.** Adding an architecture touches ≤ 6 files with no dispatcher branches; `instantiate_brain` ([`brain/arch/_registry.py:141`](../../../packages/quantum-nematode/quantumnematode/brain/arch/_registry.py#L141)) and the registry-derived `BRAIN_CONFIG_MAP` carry dispatch. The one sanctioned per-arch switch is `_build_infra_kwargs` ([`utils/brain_factory.py:54`](../../../packages/quantum-nematode/quantumnematode/utils/brain_factory.py#L54)); the one stray per-arch branch in an entrypoint is the CLI-default `match brain_type` in [`scripts/run_simulation.py:286-298`](../../../scripts/run_simulation.py#L286) (no-YAML path only).

Three of the deferred Decision-4-mechanism choices were settled at the 2026-06-05 T5 planning checkpoint: continuous-action parameterisation = tanh-squashed Gaussian; refactor scope = shared policy module unifying discrete + continuous; Gate-2 parity vehicle = a Transformer/attention architecture.

## Goals / Non-Goals

**Goals:**

- A `Continuous2DEnvironment` selectable via a config discriminator, with the grid env (`DynamicForagingEnvironment`) untouched so the T4-vs-T7 delta baseline is stable.
- Continuous-action (tanh-squashed Gaussian) heads on the four MUST PPO brains and a continuous-output adapter on the connectome brain, behind a **shared policy module** that also subsumes the existing discrete path.
- A continuous action contract consumed generically by the agent/sim loop — zero per-architecture branches added.
- Gate 2 closed: a new Transformer architecture added in ≤ 6 files with no per-architecture branches (G2.b/G2.c), engineer-hours recorded (G2.a), and a continuous-substrate floor check (G2.d).
- A migration-regression bar proving the discrete-path unification didn't change the working T4 brains.

**Non-Goals:**

- Rung-2 chemical-gradient fidelity, dynamic diffusion, and the adaptive-threshold/biphasic chemosensory sensor — **all T6** (descoped from T5 at the 2026-06-04 checkpoint).
- Real-worm validation and the L2 re-run on the upgraded substrate — **T7**.
- Native body mechanics (undulation, omega turns, pirouettes) and 3D — Future Directions / c302-Sibernetic export.
- An apples-to-apples upgraded-substrate ranking — that is T7. T5's floor check only verifies the substrate change didn't break training.
- Continuous-action adaptation of every SHOULD/MAY architecture — opportunistic, not gating.

## Decisions

### D1: Continuous-2D as a new env subclass behind a config discriminator (grid env untouched)

Add `Continuous2DEnvironment` alongside `DynamicForagingEnvironment`, selected by a new `EnvironmentConfig.env_type` discriminator (`"grid"` default | `"continuous_2d"`). `create_env_from_config` dispatches on it; the `EnvironmentType` alias generalises to the base/union; the other direct construction sites are routed through the factory. **Why:** the grid env must remain byte-stable for the RQ5 env-upgrade delta; a sibling + switch is the smallest change that guarantees that.

**Subclass-parent refinement (implementation, 2026-06-05): `Continuous2DEnvironment` subclasses `DynamicForagingEnvironment`, NOT `BaseEnvironment`.** The planning docs assumed `BaseEnvironment`, but the audit confirmed the sensing/source/state machinery (food/predator gradients, temperature/oxygen fields, `get_state`, klinotaxis, STAM) is already continuous-math and position-type-agnostic — subclassing `BaseEnvironment` would force reimplementing all of it (huge duplication + divergence risk). Subclassing `DynamicForagingEnvironment` reuses it and overrides ONLY the grid-coupled methods actually implemented in this iteration: kinematic `(speed, turn)` movement, capture-radius food consumption, and the Euclidean (vs Manhattan) distances. **Continuous (float) source placement is NOT implemented yet — it stays deferred per tasks.md §3.4**: the environment currently keeps food/predator sources on the **integer lattice within the continuous arena** (inherited placement), while only the worm position, movement, capture, and distances are continuous. Float positions are carried by an **additive optional** `AgentState.pos_continuous: tuple[float, float] | None` (+ `heading_rad: float = 0.0`) — both implemented — NOT by relaxing the existing `position: tuple[int, int]` annotation — relaxing it would cascade pyright errors through every grid consumer that passes `position` to an int-typed API (readiness finding 2026-06-06); float *source* placement is the part still deferred because it would ripple the same way through `self.foods: list[tuple[int, int]]`. The continuous env holds the float truth in `pos_continuous`, syncs a rounded int `position` as a discretized view for any inherited reader, and the grid env's int type contract stays byte-stable. For continuous, the worm is modelled as a point (single-position body) to sidestep grid body-following; rendering is headless-first (the continuous renderer is the deferred T6.render work). **Alternatives rejected:** `BaseEnvironment` parent (massive duplication); mutating `DynamicForagingEnvironment` to be coordinate-mode-parametric (entangles two substrates in one class, risks regressing the T4 baseline). The cleaner long-term design — decomposing the god-class into shared source/sensing/state + composable substrate strategies (movement/capture/placement/distance) — is a large, byte-stability-risking refactor out of T5 scope, tracked for a future change (best triggered by a 3rd substrate) in [GitHub issue #206](https://github.com/SyntheticBrains/nematode/issues/206). The §3.3 override seams are its natural extraction points.

### D2: Physical scales

Anchor to plate-arena biology: worm body length ≈ 1 mm; plate ≈ 30–50 mm; world bounds expressed in mm. Per-step forward displacement and the klinotaxis lateral head-sweep amplitude are scaled to a fraction of body length (sweep ≈ ½ body length; step ≈ body-length-per-timestep order), capture radius ≈ a small multiple of body length. Exact constants live in config (D8) with documented defaults; they are tuning parameters, not load-bearing claims. **Why:** the field's reference data (Bargmann chemotaxis, Pierce-Shimomura kinematics) is plate-scale 2D; matching geometry is what makes T7's behavioural validation defensible.

### D3: Continuous env semantics — the three grid-identity behaviours that must change

The sensing stack is already continuous, so the lift is narrow and concentrated in three grid-cell-identity behaviours plus rendering:

1. **Food capture: cell-equality → capture radius.** Replace `tuple(position) in self.foods` / `.remove` ([`env/env.py:2274,2293`](../../../packages/quantum-nematode/quantumnematode/env/env.py#L2274)) with a Euclidean within-radius test. This is the single biggest behavioural change.
2. **Source placement: integer sampling → continuous uniform-over-bounds — DEFERRED, not in T5 scope (see tasks.md §3.4).** Float food/predator placement would change only candidate generation (`rng.integers(grid_size)` at [`env/env.py:1520`](../../../packages/quantum-nematode/quantumnematode/env/env.py#L1520); validity checks are already Euclidean), but is **held back to a future iteration** to avoid the `self.foods: list[tuple[int, int]]` type ripple — the same class of change as the position-type refinement. In T5 the worm, movement, capture, and distances are continuous while **sources stay on the integer lattice within the continuous arena** (matching the Continuous2DEnvironment description above and the spec's lattice-now wording).
3. **Distances: Manhattan → Euclidean** for pheromone ([`env/pheromone.py:150`](../../../packages/quantum-nematode/quantumnematode/env/pheromone.py#L150)), predator-mechano contact ([`agent/agent.py:196`](../../../packages/quantum-nematode/quantumnematode/agent/agent.py#L196)), and nearest-food ([`env/env.py:2331`](../../../packages/quantum-nematode/quantumnematode/env/env.py#L2331)) on the continuous substrate.

Also: klinotaxis lateral offsets become physically-scaled floats (drop the fixed ±1-cell `_compute_lateral_offsets` and the `int(...)` position cast at [`agent/agent.py:858`](../../../packages/quantum-nematode/quantumnematode/agent/agent.py#L858)); analytic temperature/oxygen gradients are preferred over the integer central-difference path on the continuous env. Rendering is **headless-first** (batch runs use `--theme headless`); the TUI/pygame continuous renderer is minimal/deferred and is not a Gate-2 concern.

### D4: Continuous action contract — extend `ActionData`, consume generically (the Gate-2 lever)

The principal Gate-2 parity risk is that a continuous `(speed, turn)` action makes the agent/sim loop branch on brain type. Decision: extend the action carrier with an **optional continuous vector** field (`ActionData` gains `continuous: tuple[float, float] | None`, with `action`/`Action` retained for the discrete substrate) and teach the agent's action-application path to dispatch on the **env's action mode** (continuous-2D vs grid), *not* on the brain class. Brains emit whichever the active substrate expects via the shared policy module (D6). **Why:** keeps the one decision point in the environment (which already owns movement semantics), so no per-architecture branch is added anywhere — this is what makes G2.c (no per-arch branches) survivable. The dispatch point is the env movement method the runner calls (`Continuous2DEnvironment` overrides `BaseEnvironment._apply_movement`/`move_agent` with a `(speed, turn)` kinematic update; the discrete `move_agent(action: Action)` signature must generalise to carry the continuous vector). **Two action-consumption surfaces must both be covered**, not just the primary one: the single-action runner path (`runners.py:720`) AND the many-worlds runner-up path (`runners.py:983-1010`, plus `.action.value` logging at `:802`) — the latter consumes the discrete `.action` directly and must either route through the same mode dispatch or be scoped discrete-only on the continuous substrate. **Alternative rejected:** a parallel `ContinuousActionData` type threaded through the runner — doubles the action plumbing and invites `isinstance(brain, …)` branches.

**Dispatch signal + reset/copy seam (§1.3 readiness finding, 2026-06-06):** the runner/agent dispatch via `isinstance(Continuous2DEnvironment)` — env-driven, not brain-coupled (a `continuous_actions` bool property was considered but dropped as dead code: isinstance is needed anyway to narrow for the continuous methods). The discrete-action fallback (a discrete brain on a continuous env, before continuous heads) stays coherent because `Continuous2DEnvironment._apply_movement` re-syncs `pos_continuous` from the integer move. Two inherited methods hardcode `DynamicForagingEnvironment(...)` and so silently drop the continuous type/params: `agent.reset_environment()` (recreates the env between runs — would revert continuous→grid after run 1) and `env.copy()` (many-worlds). §1.3 must recreate the *same* env type with `Continuous2DParams` (polymorphic recreate, or an isinstance branch keeping the grid path byte-identical); many-worlds-on-continuous (which needs the `copy()` override) is guarded as not-yet-supported, single-world first.

### D5: Tanh-squashed Gaussian, action dim = 2

Continuous action = `(speed ∈ [0, max], turn ∈ [−π, π])`. Sample from a diagonal `Normal(mean, std)`, squash with `tanh`, affine-rescale to the action ranges, and apply the **log-det-Jacobian correction** to log-prob; entropy via the base-Normal entropy (with the standard squash caveat). `std` is a state-dependent head output (softplus) or a state-independent learnable log-std — chosen per brain, documented. **Why:** bounded continuous control with unbiased gradients at the bounds (SAC/PPO-continuous standard). **Alternatives considered:** Beta (naturally bounded, no Jacobian, but a less standard PPO fit and a 2-param-per-dim head) and raw Gaussian+clip (simplest but biased log-probs/gradients at the bounds) — both rejected for the headline; Beta may appear as a documented ablation if tanh-Gaussian shows boundary pathologies.

**Implementation refinements (2026-06-06):**

- **Scaling lives in the environment, not the policy (deviation from "affine-rescale to the action ranges" above).** Brains emit a *normalized* action (`speed ∈ [0, 1]`, `turn ∈ [-1, 1]`) via the tanh-squash + a fixed normalized affine; `Continuous2DEnvironment.move_agent_normalized` applies the physical rescale (`× max_step_mm`, `× π`). This keeps brains env-scale-agnostic (brain construction has no env handle) and the physical scale where movement semantics already live (D4). It is PPO-equivalent: the physical affine is a constant log-det offset that cancels in the importance ratio (same action, new/old params) and does not touch the base-Normal entropy, so re-scoring in the normalized space is exact. Cost: reported log-prob/entropy are in normalized units, not physical mm/rad.
- **Std head form = state-independent learnable `log_std`** (one `nn.Parameter` per action dim), the simplest stable PPO-continuous choice; recorded here per the "chosen per brain, documented" note. (A state-dependent softplus head remains available if a brain needs input-conditional exploration.)
- **Shared plumbing:** `_policy.py` gains `continuous_sample_tanh_gaussian` / `continuous_evaluate_tanh_gaussian` / `clamp_continuous_log_std` (stable log-det-Jacobian `2·(log2 − u − softplus(−2u))`); the rollout stores the pre-squash sample `u` (policy-independent given the action) so the update re-scores without an unstable `atanh`. The shared `RolloutBuffer` carries a `continuous_actions` flag; `ActionData.action` is relaxed to `Action | None` (continuous carries `continuous=(speed, turn)`).

### D6: Shared policy module unifying discrete + continuous (with a migration-regression bar)

Introduce `brain/arch/_policy.py` (a helper/mixin, since no base class exists) exposing one interface for: build-head, sample (`Categorical` for discrete | tanh-Gaussian for continuous), `log_prob`, `entropy`, and the PPO surrogate terms. Migrate **both** the existing discrete logic and the new continuous logic of all four MUST brains onto it (the user-chosen "unify discrete path too" scope). **Why:** the discrete logic is currently copy-pasted 4× — unifying removes drift, makes the continuous head a one-line mode switch per brain, and concentrates the parity-sensitive code in one reviewed place. **Cost / mitigation:** this touches the working T4 brains, so it ships under a **migration-regression bar** (T2/Phase-5 M1 precedent): byte-equivalence — or a declared, up-front seeded-RNG tolerance — on one discrete smoke config per migrated brain, pre/post refactor.

**Migration approach per brain (settled 2026-06-05, "Option B" — empirically measured):** MLP-PPO and connectome-PPO already use `torch.distributions.Categorical`, so they migrate onto the shared torch helpers **byte-exactly** (verified: MLP-PPO golden snapshot bitwise-identical pre/post). LSTM-PPO and CfC-PPO instead sample via `numpy rng.choice` + a *manual* `log(softmax)` / `−Σp·log p`; their migration **keeps the numpy sampler verbatim** (so the sampled-action trajectory is byte-identical) and routes only the log-probability, entropy, and surrogate through the shared torch helpers. Measured deviation on the *taken* action's log-prob (the only one PPO evaluates) is **~1e-7 (float32 round-off) at all policy-confidence levels** — entropy ~2e-7 — so the trajectory is preserved and final metrics are indistinguishable. The manual `log(softmax)` is in fact *less* numerically stable than torch's `log_softmax` for low-probability actions, so this is a slight stability improvement, not a regression. This is a declared tolerance of ~1e-6 for LSTM/CfC — recorded here and in the T5 logbook — and does **not** weaken the byte-equivalence claim in any way that affects results (it is not the forbidden "different-seed run" that fully swapping the sampler would cause). MLP-PPO + connectome-PPO remain byte-exact. **Alternative rejected:** per-brain inline continuous logic (mirrors today's copy-paste; 4× drift and parity-leak surface) and helper-for-continuous-only (leaves the discrete copy-paste in place).

**Action-dim plumbing (parity-preferred):** `_build_infra_kwargs` hard-codes `num_actions: 4` in every branch ([`utils/brain_factory.py:117-157`](../../../packages/quantum-nematode/quantumnematode/utils/brain_factory.py#L117)). The continuous action dimension (2) and mode are carried in `BrainConfig` (the continuous-action config fields), NOT added as a new `_build_infra_kwargs` per-arch branch — so the action mode flows through config, the `num_actions=4` infra-kwarg is interpreted as the discrete action-set size and ignored in continuous mode, and `_build_infra_kwargs` (the one sanctioned per-arch switch) gains no continuous-specific branch. This keeps the continuous refactor off the parity-sensitive switch.

### D7: Connectome continuous-output adapter — readout swap only, strict-mask preserved

The connectome motor-pool readout (`_pool_motor` → `4×4 readout → Categorical`, [`connectome_ppo.py:548,393,705`](../../../packages/quantum-nematode/quantumnematode/brain/arch/connectome_ppo.py#L548)) gains a `4 → (mean, log_std)` continuous head; sampling routes through the shared module (D6). **The chemical-synapse strict-mask and fixed gap junctions are untouched** — they constrain the internal `w_chem` recurrence ([`connectome_ppo.py:697,1358-1362`](../../../packages/quantum-nematode/quantumnematode/brain/arch/connectome_ppo.py#L697)), entirely upstream of the readout, and are orthogonal to output type. **This retires the roadmap's stated T5 failure mode** ("continuous-output adapter incompatible with the strict-mask"): the audit shows no such incompatibility. The residual connectome risk is *capacity* — a 4-motor-pool → 2-D continuous head is narrow; a per-motor-neuron (un-pooled) readout is a documented fallback ablation if the pooled head underperforms the G2.d floor.

**Implementation note (2026-06-06):** the readout *matrix* is reshaped `(4,4) → (2,4)` so it maps the 4 motor classes to the 2-D Gaussian **mean**; the `log_std` is a **separate state-independent learnable parameter** on the `Topology` (uniform with the other MUST brains' continuous heads), NOT produced by the readout — so the "`4 → (mean, log_std)`" phrasing above means "a continuous head" rather than a single 4→(2+2) matrix. `log_std` rides the `Topology` `nn.Module`, so it persists in `state_dict` exactly like `readout`. The strict-mask invariant is now **test-covered** (`test_connectome_ppo_continuous.py`): `m_chem` / `g_gap` / initial `w_chem` are byte-identical across the discrete and continuous output modes, and non-wild-type edges stay 0 after continuous PPO.

### D8: Transformer arch as the Gate-2 parity vehicle; refactor the CLI-default branch

Add a Transformer/attention PPO brain (`transformer-brain`) through the plugin interface as the **G2.b/G2.c verification** (files-touched ≤ 6; no per-arch branches), recording engineer-hours for G2.a. It doubles as the roadmap's MAY transformer comparison row. To make the no-per-arch-branch claim hold at the entrypoint too, refactor the `match brain_type` CLI-default block ([`scripts/run_simulation.py:286-298`](../../../scripts/run_simulation.py#L286)) into a registry-driven default. **This refactor ships in a SEPARATE prior commit** — it is a one-time platform fix, not a per-architecture cost — so it is NOT counted in the transformer-addition `git diff --name-only` that measures G2.b. **Files-budget measurement (reconciling the two enumerations):** G2.b's threshold is `≤ 6 files`; its parenthetical list in [`phase6-tracking` design.md Decision 6](../phase6-tracking/design.md) ("registry registration + brain implementation + config class + config example + smoke test + docs") is illustrative. The *operative* enumeration is the [plugin-developer-guide](../../../docs/architecture/plugin-developer-guide.md) budget: new module (brain + config class + `@register_brain`), `dtypes.py` enum member, `brain/arch/__init__.py` import/export, `config_loader.py` `BrainConfigType` union, `brain_factory.py` `_build_infra_kwargs` branch *only if non-default*, and a test file = ≤ 6, with the YAML config example explicitly *outside* the 6 — as are the cross-repo brain-arch enumeration updates (`AGENTS.md`, `README.md`, `CONTRIBUTING.md`, `openspec/config.yaml`, currently "24 architectures"), which are required for merge but are platform-bookkeeping, not per-architecture parity cost (landed in a separate commit from the runtime wiring so the G2.b `git diff` stays clean). The ≤ 6 threshold itself is unchanged (this is a pre-fire measurement clarification, not a gate recalibration). **Why a real arch over a hypothetical baseline:** the user chose the transformer to get a reusable comparator; G2.a explicitly makes engineer-hours non-load-bearing precisely so a heavier arch doesn't distort the parity verdict. **Gate-2 floor check (G2.d):** connectome + MLP-PPO train cleanly on continuous-2D klinotaxis to ≥ 50 % of their T4 grid-substrate mean episode return (the 50 % floor is wide because discrete-grid and continuous-2D return scales aren't directly comparable; this checks "didn't break", not ranking).

### D9: Gate-2 decision recording

The Gate-2 GO/PIVOT/STOP decision is written in the **T5 logbook** (not this tasks.md, per Decision 6), evaluated against G2.a–G2.d. PIVOT trigger: G2.b OR G2.c fails → L1 refactor pivot (+2–4 weeks). STOP trigger: post-pivot the interface is still fundamentally incompatible with a MUST family. The `phase6-tracking` tracker's Gate-2 checkboxes link to the decision once it lands.

## Risks / Trade-offs

- **[Discrete-path unification regresses the working T4 brains]** → D6 migration-regression bar; MLP/LSTM byte-equivalence mandatory, CfC/connectome declared tolerance; run before any continuous work merges.
- **[Continuous action contract leaks a per-arch branch and fails G2.c]** → D4 routes the one decision point through the env action-mode, not the brain class; the plugin-parity test asserts no `isinstance(brain, …)` / per-arch branch in agent + sim/training loops.
- **[Tanh-Gaussian PPO instability or boundary saturation]** → standard mitigations (log-std clamping, Jacobian-corrected log-prob, entropy bonus); G2.d floor check catches "broke training"; Beta head available as a documented ablation.
- **[Connectome pooled 4→2 head too low-capacity]** → per-motor-neuron readout fallback ablation (D7); strict-mask itself is de-risked.
- **[Scope creep — T6 env-fidelity bleeds into T5]** → explicit non-goal; T6 owns the adaptive sensor + dynamic gradients; T5 ships static-geometry continuous coords only.
- **[Continuous food-capture changes difficulty vs grid, confounding the floor check]** → G2.d is a wide ≥50 % "didn't break" floor, not a ranking; capture radius is a documented, tunable config default.
- **[Rendering/viewport assume a discrete grid]** → headless-first; continuous TUI/pygame is minimal/deferred, out of Gate-2 scope.

## Migration Plan

1. Action contract + shared policy module (`_policy.py`) with discrete parity first; migrate the four MUST brains onto the discrete path; **run the migration-regression bar** (gate before proceeding).
2. Continuous-2D env subclass + config discriminator + factory dispatch; grid env untouched; continuous env unit tests (capture radius, kinematics, Euclidean fields).
3. Continuous tanh-Gaussian heads on the four MUST brains via the shared module; connectome continuous-output adapter; continuous-substrate smoke training.
4. Transformer arch through the plugin interface; refactor the CLI-default branch; run the plugin-parity test (≤6 files, no branches).
5. G2.d floor check (connectome + MLP on continuous klinotaxis ≥ 50 % of T4 baseline); analysis (platform-refactor delta); T5 logbook + Gate-2 decision; update `phase6-tracking` tasks.md + roadmap T5 row.

**Rollback:** the env-type discriminator defaults to `"grid"`; if continuous training proves unworkable, the discrete substrate and brains remain fully functional and the change degrades to "shared policy module + transformer arch" without the continuous env — but Gate 2's G2.d would then be unmet and the L1-refactor pivot triggers.

## Open Questions

- **Std head form per brain** (state-dependent softplus vs state-independent learnable log-std) — resolve during D6 implementation; record per brain.
- **Connectome readout granularity** (pooled 4-vector vs per-motor-neuron) — default pooled; switch to per-neuron only if the G2.d floor fails.
- **Capture radius / step length / sweep amplitude exact defaults** — set during D2/D3 implementation against a klinotaxis smoke; documented in the continuous scenario YAML.
- **Whether the Transformer also gets a continuous head in T5 or discrete-only for the parity test** — discrete-only is sufficient for G2.b/G2.c; a continuous head is opportunistic.
