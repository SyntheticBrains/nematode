## ADDED Requirements

### Requirement: Phenotypic Cycling Metric

The system SHALL provide a `phenotypic_cycling` pure function that detects oscillation in a per-generation series via OLS-detrended autocorrelation peak detection within a bounded lag range, with a permutation-null p-value.

**Implementation note (non-normative):** the shipped function tracks the peak POSITIVE autocorrelation (period repetition) rather than peak `|correlation|`, since for a pure sine of period P the autocorrelation is `cos(2π·lag/P)` — `|corr|` peaks at half-period (anti-phase), not the true period. Reporting peak positive correlation gives the "dominant period" semantics the spec scenarios test against. Earlier drafts of this spec mentioned Lomb-Scargle and FFT-bin-power thresholds; the shipped implementation uses neither (autocorrelation suffices for the verdict-gate inference task and avoids a scipy dependency).

#### Scenario: Detects Pure Sine Cycling

- **GIVEN** a per-generation series of length ≥ 30 generated by a pure sine wave with period in [3, 15] generations
- **WHEN** `phenotypic_cycling(series, lag_range=(3, 15))` runs
- **THEN** the returned dict SHALL contain `cycling_detected=True`
- **AND** the dominant period SHALL match the input wave's period within ±1 generation (peak POSITIVE autocorrelation; not peak `|correlation|` which would land at half-period for a pure sine)

#### Scenario: Rejects Pure Linear Trend

- **GIVEN** a per-generation series consisting only of a strictly monotone linear ramp (no periodic component)
- **WHEN** `phenotypic_cycling(series)` runs
- **THEN** the returned dict SHALL contain `cycling_detected=False`
- **AND** the function SHALL OLS-detrend the series before computing autocorrelation; pure-linear residuals (ratio < 1e-12 of original variance) SHALL trigger an early-return with `dominant_period=None` and `p_value=NaN` rather than falsely flagging cycling at the lowest lag

#### Scenario: Statistical Significance

- **GIVEN** a per-generation series with a clear periodic component
- **WHEN** `phenotypic_cycling(series)` runs
- **THEN** the returned dict SHALL contain a `p_value` field derived from a permutation null distribution (typical 500-shuffle Monte Carlo with the standard `(1+k)/(1+N)` correction so finite-N p-values are never exactly zero)
- **AND** detection SHALL require `p_value < 0.05`

### Requirement: Trait Escalation Metric

The system SHALL provide a `trait_escalation` pure function that fits a closed-form OLS regression over a per-generation trait-mean series within a configured `gen_window` and reports slope, slope sign, slope SE, and a two-sided p-value.

**Implementation note (non-normative):** the shipped function uses an asymptotic standard-normal CDF for the p-value rather than a t-distribution CDF, since (a) `n` in the default window (gens 5..30, n up to 26) is large enough that the difference between z and t is small (~3% on p), and (b) avoiding scipy keeps the dependency footprint trivial. The verdict gate's `|slope| / SE > 2.0` threshold is unaffected by this choice. The aggregator (PR 5) can report the t-statistic directly if a reviewer wants tighter inference. Zero-residual perfect-fit cases (slope is exactly known) return `p_value=0` if the slope is non-zero and `p_value=1` if zero, instead of the degenerate "unknown" path.

#### Scenario: Detects Monotone Increase

- **GIVEN** a per-generation series with a clear monotone-increasing trend (e.g. linear ramp + small noise)
- **WHEN** `trait_escalation(values, gen_window=(5, 30))` runs
- **THEN** the returned dict SHALL contain `escalation_detected=True`
- **AND** `slope_sign=+1`
- **AND** `p_value < 0.05`

#### Scenario: Rejects Flat Series

- **GIVEN** a per-generation series with no trend (zero-mean noise around a constant)
- **WHEN** `trait_escalation(values)` runs
- **THEN** the returned dict SHALL contain `escalation_detected=False`

#### Scenario: Window Skips Bootstrap Noise

- **GIVEN** a series whose first 5 generations are noisy bootstrap and whose remaining generations show a clear trend
- **WHEN** `trait_escalation(values, gen_window=(5, 30))` runs with the default window
- **THEN** the regression SHALL be fit over generations 5..30 only
- **AND** the bootstrap-noise generations SHALL NOT influence the slope estimate

### Requirement: Fitness Lag Metric

The system SHALL provide a `fitness_lag` pure function that computes the cross-correlation between two per-generation fitness series (prey and predator) and reports the lag at which the **positive** correlation peaks. Anti-phase coupling (where the two series are inversely related) is captured separately by `coupled_rate`; tracking peak `|correlation|` here would conflate "where do these align?" with "where are they anti-aligned?" — for periodic inputs `|corr|` peaks at half-period regardless of whether the series actually align.

#### Scenario: Detects Phase-Shifted Coupling

- **GIVEN** two per-generation series constructed as `prey[g] = sin(g)` and `predator[g] = sin(g + lag)` with a known constructed lag (such that `predator` trails `prey` by `lag` generations)
- **WHEN** `fitness_lag(prey, predator, max_lag=20)` runs
- **THEN** the returned best-lag SHALL match the constructed lag within ±1 generation (peak POSITIVE correlation; the implementation reports the lag at which the two series align in phase)

#### Scenario: Returns NaN For Trivial Series

- **GIVEN** two zero-variance series (constants)
- **WHEN** `fitness_lag` runs
- **THEN** the returned best-lag SHALL be `NaN` (cross-correlation undefined)

### Requirement: Coupled Rate Metric

The system SHALL provide a `coupled_rate` pure function that measures the covariance between per-generation rates of change of two trait series, indicating whether the two sides change in lock-step or independently.

#### Scenario: High Coupling Detected

- **GIVEN** two trait series where each generation's delta in one series equals the delta in the other (perfect coupling)
- **WHEN** `coupled_rate(trait_a, trait_b)` runs
- **THEN** the returned coupling coefficient SHALL be near +1 (correlation of deltas)

#### Scenario: Independent Series

- **GIVEN** two trait series with statistically independent generation-to-generation deltas
- **WHEN** `coupled_rate` runs
- **THEN** the returned coupling coefficient SHALL be near 0

### Requirement: Generality Metric

The system SHALL provide a `generality` pure function that summarises the held-out opponent fitness curve and emits a single scalar indicating whether elite performance generalises across opponents.

#### Scenario: High Generality (Improvement Across All Opponents)

- **GIVEN** a held-out probe series where the elite's fitness improves over generations against ALL held-out opponents
- **WHEN** `generality(probe_series)` runs
- **THEN** the returned generality scalar SHALL be near +1 (uniform improvement)

#### Scenario: Self-Play Overfitting (Decline On Held-Out)

- **GIVEN** a held-out probe series where elite fitness on training opponents climbs but fitness on held-out opponents declines or stays flat
- **WHEN** `generality(probe_series)` runs
- **THEN** the returned generality scalar SHALL be near 0 or negative
- **Note:** an aggregator-level "self-play overfitting suspected" flag is a non-normative reporting concern; when a generality threshold is calibrated from pilot evidence, the aggregator's `summary.md` MAY add a flag derived from the generality scalar. The threshold value is not pinned by this spec. (Historical: the probe body shipped a NaN placeholder in early PR commits before being wired in PR 5b — see "NaN-Tolerance For Empty Held-Out" scenario for the residual NaN handling that remains relevant for missing-bundle cases.)

### Requirement: Aggregator Verdict Output

The system SHALL provide an aggregator that consumes per-seed Red Queen metric series and produces a verdict CSV plus a human-readable summary.

#### Scenario: Verdict CSV Schema

- **GIVEN** completed full-run results for N seeds
- **WHEN** the aggregator runs
- **THEN** it SHALL produce `verdict.csv` with one row per seed and the following columns in this order (column order is normative so machine-readers can rely on positional access):
  1. `seed: int`
  2. `cycling_detected: bool`
  3. `escalation_detected: bool`
  4. `escalation_slope: float`
  5. `escalation_p_value: float`
  6. `cycling_period: int | NaN`
  7. `cycling_p_value: float`
  8. `generality_scalar: float`
  9. `gate_fires: bool` (cycling OR escalation)
- **AND** the verdict CSV SHALL be readable by `pandas.read_csv` without dtype coercion
- **AND** when both prey and predator sides have firing metric values for the same seed, the smaller-`p_value` side's slope/period/p-values SHALL be written (more decisive). When neither side has computed values, NaN-shaped placeholders are written. Generality scalar prefers the prey side; falls back to the predator side when prey is NaN

#### Scenario: Summary Markdown Output

- **GIVEN** completed full-run aggregation
- **WHEN** the aggregator runs
- **THEN** it SHALL produce `summary.md` containing:
  - The overall verdict (`GO` / `STOP` / `PIVOT` / `INCONCLUSIVE` per the co-evolution capability's "Verdict Aggregation Across Seeds" scenario) and the seeds-firing count
  - Per-seed cycling and escalation results
  - The generality probe trajectory plot reference
  - A clear note that generality is reported but not a verdict input
  - For M5.7, a Baldwin-readout block clearly labelled "secondary observation, not a verdict input"

#### Scenario: Input Layout Auto-Discovery

- **GIVEN** a `--root` argument pointing at one of three layouts:
  1. A campaign root containing per-seed subdirs (`<root>/seed-<N>/<session_id>/...`).
  2. A per-seed parent dir containing one or more session subdirs (`<root>/<session_id>/...`) with no `seed-<N>/` wrapping — produced by the smoke driver and by single-seed manual runs.
  3. `<root>` itself IS a session dir (contains `prey/`, `predator/`, `champion_history.json` directly).
- **WHEN** the aggregator runs
- **THEN** it SHALL auto-discover the layout: prefer (1) when any `seed-<N>/` subdir is present; otherwise fall back to (2) or (3) and label the synthetic seed id as `-1` in the verdict CSV + summary
- **AND** for layout (2), when multiple session subdirs exist the most-recently-modified is selected by default; an explicit `--session <id>` SHALL override the most-recent default
- **Rationale:** layout (1) is the production pilot/full layout; layouts (2) and (3) are smoke-driver and ad-hoc convenience cases. Pinning all three in the spec lets `tasks.md` 8.5's "smoke-validate the aggregator" exercise be deterministic without per-script branch logic in the test harness.

#### Scenario: NaN-Tolerance For Empty Held-Out

- **GIVEN** an aggregator input where `generality_probe.csv`'s `fitness` column is empty or partially NaN (e.g., the prey reference bundle was missing at `__init__` — see the co-evolution capability's "Held-Out Bundle Missing Falls Back To No-Op" scenario — so the predator-side probe was a no-op for the run; or the side's `champion_history` was empty for early generations so the probe wrote no rows for that side)
- **WHEN** the aggregator computes per-seed metrics
- **THEN** the generality scalar SHALL be reported as NaN (not 0.0, not "skip"), surfaced as `N/A` in `summary.md` and the empty string in `verdict.csv`
- **AND** the per-seed gate firing SHALL be unaffected (verdict gate uses cycling + escalation only per design.md D6); generality is interpretation-only
- **Historical note:** prior to PR 5b, `_probe_one_opponent` returned NaN unconditionally (the deferred-body design); the aggregator's NaN-tolerance was originally written for that case. PR 5b wired the body (per "Probe Held-Out Wiring (Option B)") so the NaN path now surfaces only via the empty/missing held-out cases described above.
